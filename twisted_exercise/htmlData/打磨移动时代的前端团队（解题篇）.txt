<script type="text/javascript">
    $(function()
    {
        //取消背景层的滚动条滚动样式
        /*$(document).css(
            {"overflow":"none","overflow":"hidden","_overflow":"hidden","padding":"0 17px 0 0"}
        );*/

        $('#accept').removeAttr('checked');
        
        //申请创建主题弹框
        $('body').delegate('.apply_CreatF', 'click', function()
        {
            var login_username = '';
            var cookie = document.cookie.split(";");
            for (var i in cookie){
                var val = cookie[i];
				var tmp = val.split('=');
				if (tmp[0].trim() == 'UserName'){
					login_username = tmp[1];
				}
            }
        	if (login_username == ''){
				location.href = "https://passport.csdn.net/account/login";
            }
        	$(".mark_apply_create").appendTo('body').removeClass("hide").fadeIn(100);
        	$(".mark_apply_create").find(".popup_c:eq(0)").show().siblings().hide();

        });

        $("body").delegate('#accept', 'click', function()
        {
            checkState();
            document.documentElement.style.cssText = 'overflow:none;+overflow:hidden;_overflow:hidden;';
            document.body.style.cssText = 'overflow:hidden;+overflow:none;_overflow:none;padding:0 17px 0 0;';
        });

        //提交创建主题
        $('body').delegate('.create_forum_submit', 'click', function(){
            var tel = $(".popup_list input[type='tel']").val();
            if($('.input_forum_name').val().trim() == ""){
                $('.create_forum_error').html('请填写主题名称').show();
                $('.input_forum_name').focus();
            }else if($('.input_forum_describe').val().trim() == ""){
                $('.create_forum_error').html('请填写主题描述').show();
                $('.input_forum_describe').focus();
            //}else if(/^13\d{9}$/g.test(tel)||(/^15[8,9]\d{8}$/g.test(tel)||(/^15[8,9]\d{8}$/g.test(tel))
            }else if(/^1[3||5||8]\d{9}$/g.test(tel))
            {
                $(this).attr("disabled", "disabled");
                var data = $('.create_forum_form').serialize();
                $.post("/forum/create", data, function(result){
                    if(result.status){
                        $('.create_forum_error').hide();
                        $(".mark_apply_create").find(".popup_c:eq(2)").show().siblings().hide();
                    } else{
                        $('.create_forum_submit').attr("disabled", false);
                        $('.create_forum_error').html(result.error).show();
                    }
                });
            }
            else
            {
                $('.create_forum_error').html('请填写正确的手机号').show();
                $(".popup_list input[type='tel']").focus();
            }
			return false;
        });
        
        //状态检查
        function checkState()
        {
            var flag = false;
            var isCheck = $("#accept").prop("checked");
            if(isCheck != null && isCheck == true)
            {
                flag = true;

            }
            if(flag)
            {
            	$('#next').removeAttr("disabled").addClass("next_02");
                $('body').delegate('#next', 'click', function()
                {
                    $(".mark_apply_create").find(".popup_c:eq(1)").show().siblings().hide();

                });
            }
            else
            {
                $('#next').attr("disabled",true).removeClass('next_02');
            }
            //alert(flag);

        };


        $("body").delegate('.close', 'click', function()
        {
        	$(".mark_apply_create").addClass("hide");
            document.documentElement.style.cssText = '';
            document.body.style.cssText = '';
        });
        
    });
</script>

                <hr><blockquote>
  <p><strong>（二）解题篇</strong></p>
</blockquote><h2>只是个开始</h2><blockquote>
  <p>此篇名为“解题”，如果您真希望通过看一篇帖子就能解决团队所有问题，那么我要让您失望了， 因为此篇并非问题的终结，而是开始。</p>
</blockquote><h3>前端团队究竟遇到了什么问题？</h3><blockquote>
  <p>看看你中枪了么</p>
</blockquote><ol>
<li>项目之初，需求不明，死线（dead line）已定。</li>
<li>需求改动频繁，排期总是被压缩。</li>
<li>总是缺人手，加班不断，且代码质量差。</li>
<li>前后端联调环境搭建耗时，且定位问题麻烦，导致联调时间不可控。</li>
<li>UI过于纠结界面细节，导致交付时间不可控。</li>
<li>移动端，设备多，兼容问题多，导致自测时间不可控。</li>
<li>领导说，PM说，UI说，UE说。信息不对等，到底听谁说？</li>
<li>基于历史原因，总是无法采用更恰当的技术方案，导致团队技术脱节。</li>
<li>一边加班，一边吐槽，一边被吐槽。</li>
<li>走人了，换了个团队，问题依旧。</li>
</ol><h3>事不关己？</h3><blockquote>
  <p>无论你是否关心以上的问题，我想说的是，我们的技术水平来源于自己的精力分配，而我们的精力分配又受限于团队任务分配，所以，每个看似是Leader的问题，其实你我都需要对自己负责。</p>
</blockquote><h3>当我说“提高前端生产力”时，我在说些什么？</h3><blockquote>
  <p>从上面的“问题堆”看到，阻碍我们项目交付的，不只是技术，这些非技术的问题更要命。</p>
</blockquote><h2><strong>一起聊聊这个“问题堆”</strong></h2><hr><p><strong>为了方便举例，我假设存在A ，B两类开发者</strong></p><ol>
<li><p><strong>项目之初，需求不明，死线（dead line）已定</strong></p>

<blockquote>
  <p>但凡有过一些工作经验的开发者，都能接受这个“残酷的事实”。 <br>
  因为他们懂得站在<strong>更高层次</strong>看待这个问题。</p>
  
  <p><strong>开发的目的有时候是为了抢占市场，所以时间点往往至关重要。</strong></p>
  
  <ul><li>接手此类项目的开发者有两种心态: <br>
  A. “压力山大”，立刻开始设计、编码。  <br>
  B. “压力山大，谋定而后动”, 基于业务，权衡利弊，发现风险，积极沟通，设计方案而后动。  <br>
   <br>
  我喜欢后者 。 做到这些要求你<strong>觉得自己真的很有必要“懂业务”</strong>。 <br>
  而你的这个选择使你成为了团队今天的<strong>骨干力量</strong>。</li></ul>
</blockquote></li>
<li><p><strong>需求改动频繁，排期总是被压缩。</strong></p>

<blockquote>
  <p><strong>需求改动错了么？</strong></p>
  
  <ul><li>需求变更通常有三种原因： <br>
  <ol><li>源于竞争态势的变化，战略的快速调整。</li>
  <li>敏捷的产品开发实践，小步快跑，渐进明细。</li>
  <li>产品设计人员没有想清楚。 <br>
  </li></ol></li>
  <li>情况1,2 无可厚非，拥抱变化不变的真理。<a href="http://baike.baidu.com/link?url=E87Vd_BR6od8DT0zcIWU3yGG4t7tVsc8mLYgWelukEGS2F7xUAqUVKxmcI1ysgedyLIxgteBBxR6eC1KRBW7Sq">敏捷开发</a>的核心，就是拥抱变化。</li>
  <li>情况3，我想替产品开发人员（PM）说句话。 <br>
  每一个产品的成败不只是取决于PM，团队中的所有人都是参与者，都要对结果负责。</li>
  <li>当面对一个经常变更需求的PM</li>
  <li>A.只会抱怨”这个PM不给力”</li>
  <li>B.眼光放在项目目标，思考自己能做些什么。 <br>
   <br>
  我想对A说: <strong>大家需要的不是你的抱怨，而是你的“补位”。</strong></li></ul>
</blockquote></li>
<li><p><strong>总是缺人手，加班不断，且代码质量差。</strong></p>

<blockquote>
  <p>缺人手其实是个相对概念。</p>
  
  <ol><li>项目确实过于紧急，基于目前的人数，无论什么样的开发者都会吃不消。</li>
  <li>业务频率正常，团队生产力低下</li></ol>
  
  <p>对于1, 暴露的问题在于内和外</p>
  
  <ul><li>内： 对业务的增量没有充分预估，做好人力储备，或技术架构对人力内调不支持。</li>
  <li>外： 缺乏跨团队借调人力的能力。 <br>
   <br>
  对于2，需要定位原因，是前端团队自身工作流问题，还是与其它团队配合问题。分而治之。此时<strong>最好不要再抱怨“缺人手”，以免更加被人“不看好”</strong>。</li></ul>
  
  <p><em>* 代码质量问题 *</em> ：我不觉得代码质量总是第一位的。代码的价值应该体现在每一次项目的成功交付。在遵守模块化开发的前提下，烂代码是可以在后续以模块的规模整体替换的。我能接受以低质量代码换取每一次迭代的快速响应。成熟团队会对重点项目进行<a href="http://baike.baidu.com/link?url=6cHGUOfqEmwSGhIADI0gkK6iYeNOC1CGCGTVvlU4yXQJrfiZC8I2XxV7QfN5eKmn6geLxlzx-kfLyoEAaJwDHq">codeReview</a>或<a href="http://baike.baidu.com/link?url=kGOL5CgRqxyI_Rt0TgtFQmv0DBqawtVeY0bejBYhsvlVjcN__P40d3mGnII3Dqkw9Elccy9MGYu_gXYwZXPJ0a">结对编程</a>或推行<a href="http://baike.baidu.com/link?url=B7ZvnooOTx75mxzphuTkM-0gCdkReYvJb2zuyvhXlz109RS8Q0tOl2cfl842d5c1kjjulD2ix2-x6KnaKrOY3upxfagX1DQ53eeCHnqEtwe">TDD</a>，逐步优化代码质量。</p>
</blockquote></li>
<li><p><strong>前后端联调环境搭建耗时，且定位问题麻烦，导致联调时间不可控。</strong></p>

<blockquote>
  <p>解决之道是“前后端分离” ，目前主流有三种做法，这取决于团队背景、前端的技术力量以及团队技术总负责人的技术选型。</p>
  
  <p><em>* 前后端分离的技术方案：*</em></p>
  
  <ol><li>前端团队掌控线上Http Server，负责View层的拼装，以及少量展现逻辑。数据在http Server端与传统后端在同一机房的网络环境下通过接口交换数据。 <br>
  <ul><li>好处： <br>
  <ol><li>后端不关心view层只做数据接口。</li>
  <li>对SEO支持交给更加了解SEO的前端负责。</li>
  <li>可以做诸如react服务端同构的技术方案(阿里已经有了最佳实践)。</li></ol></li>
  <li>弊端： <br>
  <ol><li>前端通常缺乏运维经验，线上服务崩溃风险高。</li>
  <li>上线变得更加复杂。 <br></li>
  <li>前端团队不触碰服务端事务。通过给后端提供<a href="https://github.com/ian000/gulp-turbo/wiki/%E4%BD%BF%E7%94%A8turbo%EF%BC%8C%E5%89%8D%E5%90%8E%E7%AB%AF%E5%A6%82%E4%BD%95%E8%81%94%E8%B0%83%E9%A1%B5%E9%9D%A2">初始化数据+demo html页</a>的形式进行分离。</li></ol></li>
  <li>好处: <br>
  <ol><li>后端只需要copy前端demo源码构建view层</li>
  <li>后端只做数据接口（同步接口，异步接口）</li></ol></li>
  <li>弊端： <br>
  <ol><li>后端还是需要随时根据HTML demo的变化，重新构建view</li>
  <li>后端需要管理静态资源版本 <br>
  </li>
  <li>把html放在CDN上，静态资源版本完全由前端控制，后端只做接口。 <br>
  <ul><li>好处： <br>
  <ol><li>前后端完全分离</li></ol></li>
  <li>弊端： <br>
  <ol><li>后端很难做session，所有的cookies都需要前端判定，并不适合所有页面。 <br>
  <br>
  <strong>广告</strong>：我所在的团队把2，3的前后端分离解决方案，整合到了我们的工作流里面并做了开源，旨在为初创团队持续提供完整高效的前端工作流。它是一个gulp插件，名为 <a href="https://www.npmjs.org/package/generator-turbo">turbo</a></li></ol></li></ul></li></ol></li></ul></li></ol>
</blockquote></li>
<li><p><strong>UI过于纠结界面细节，导致交付时间不可控。</strong></p>

<blockquote>
  <p>一个FE,没被UI的“像素眼”盯过,那么他的FE生涯将是不完整的。</p>
</blockquote>

<ol><li>精益求精永远是没错的，但还是要从项目的紧急程度来看问题。好的FE需要一定的沟通能力。</li>
<li>FE需要协同UI，UE 制定组件化视觉规范。</li></ol></li>
<li><p><strong>移动端，设备多，兼容问题多，导致自测时间不可控。</strong></p>

<blockquote>
  <p>移动时代虽然让FE脱离了恼人的各种IE，却又带来了更多的平台的兼容问题。我们希望通过一个个可复用，经过测试的组件来减少日常开发的跨终端的测试。 <br>
  试想，如果每次开发，80%以上的模块都是复用可靠且兼容的组件开发，那么这些兼容性问题，将会随着组件库的丰富，越来越少。</p>
</blockquote></li>
<li><p><strong>领导说，PM说，UI说，UE说。信息不对等，到底听谁说？</strong></p>

<blockquote>
  <p>毫无疑问，无论“是谁说”，最后都得听领导的，别问我为什么！ <br>
  （但又不能事事都问领导，轻重请自行把握）</p>
</blockquote>

<ol><li>唯一项目负责人原则： <br>
<ol><li>由于FE对接的层面很多，包括 PM，UE, UI，RD ,如果收到每个角色的反馈，就立即动手，后果可想而知，一个FE会不会工作，由此可鉴。</li>
<li>我推荐“唯一项目负责人”原则，FE依赖的资源都需要这个负责人审核并给出，包括MRD，UI，UE，接口文档（接口文档与RD的后续沟通可以脱离项目负责人）,可有效避免信息不对等。 如果有项目经理最好，没有项目经理 默认是 产品经理（PM）</li></ol></li></ol></li>
<li><p><strong>基于历史原因，总是无法采用更恰当的技术方案，导致团队技术脱节。</strong></p>

<blockquote>
  <p>其实架构也有很多当时的受限条件 <br>
  </p><ol><li>业务规模</li>
  <li>团队阶段</li>
  <li>时代技术背景</li>
  <li><p>架构师技术能力和视野</p></li></ol><p></p>
</blockquote>

<ul><li>要解决这个问题，首先我们必须承认，<strong>根本不存在完美的技术方案和架构</strong>。我们需要拥抱变化，为向未来的技术和架构过渡做好准备。</li>
<li><strong>渐进式架构与技术债务</strong> <br>
<ol><li>基于团队的不同形态设计适合不同阶段的架构与技术堆栈。同时，需要以开放的心态看待 “技术债务”，就像我们通常不喜欢欠债，但有时候贷款也是为了创造更大的利润。关键在于<strong>管控</strong>和<strong>权衡</strong>。</li>
<li><strong>何时可以启用新技术与架构？</strong></li>
<li>对于不重要不紧急的产品</li>
<li>如果没有，就在组内做技术驱动的技术产品，比如团队网站，以此来打磨新技术或架构。然后选择一个恰当的时机在新项目里推行。</li></ol></li></ul></li></ol><li><p>一边加班，一边吐槽，一边被吐槽。</p>

<blockquote>
